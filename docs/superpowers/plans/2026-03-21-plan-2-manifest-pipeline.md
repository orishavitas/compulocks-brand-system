# Manifest Pipeline Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the manifest pipeline that auto-extracts component metadata from Storybook stories, commits a machine-readable `component-manifest.json` to the repo root on every push via a git pre-push hook, and extends the n8n Code→Figma workflow to fetch that manifest so it is available for downstream Figma sync.

**Architecture:** A standalone ESM script (`scripts/export-manifest.mjs`) statically parses `components/**/*.stories.tsx` files with regex — no story execution, no Babel, no bundler. A git pre-push hook invokes the script, auto-stages and commits `component-manifest.json` if the content changed. n8n workflow-a gains one new `httpRequest` node that fetches the committed manifest from the GitHub Contents API using the same axios/base64 pattern already established in the Transform node.

**Tech Stack:** Node.js ESM (no new dependencies — only built-in `crypto` and `fs`), bash for the hook, n8n 2.6.4 Community with `httpRequest` typeVersion 3

### Task 2: Write `scripts/export-manifest.mjs`

**Files:**
- Create: `scripts/export-manifest.mjs`

The script uses only Node built-ins (`fs`, `path`, `crypto`, `glob` via `fs.readdirSync` recursive walk). No external dependencies. Outputs valid JSON to `component-manifest.json` at repo root.

- [ ] **Step 1: Create `scripts/export-manifest.mjs`**

```javascript
#!/usr/bin/env node
/**
 * export-manifest.mjs
 *
 * Statically parses components/**\/*.stories.tsx files to extract:
 *   - Component name (from default export `title` field, last segment after '/')
 *   - Variant names (named const exports, excluding 'default')
 *   - State names (keys of the `argTypes` block that are not 'variant')
 *   - SHA-1 hash of name + sorted variants + sorted states
 *
 * Writes component-manifest.json to repo root.
 *
 * Usage:
 *   node scripts/export-manifest.mjs
 *   node scripts/export-manifest.mjs --dry-run   # prints JSON, does not write
 */

import { readFileSync, writeFileSync, readdirSync, statSync } from 'node:fs';
import { join, resolve, dirname } from 'node:path';
import { fileURLToPath } from 'node:url';
import { createHash } from 'node:crypto';

const __dirname = dirname(fileURLToPath(import.meta.url));
const REPO_ROOT = resolve(__dirname, '..');
const COMPONENTS_DIR = join(REPO_ROOT, 'components');
const OUTPUT_PATH = join(REPO_ROOT, 'component-manifest.json');
const DRY_RUN = process.argv.includes('--dry-run');

// ── Parser functions (exported for tests) ──────────────────────────────────

/**
 * Extracts the title from a default export object: title: 'Components/Button'
 * Returns the full title string (e.g. 'Components/Button'), or null if absent.
 */
export function extractTitle(source) {
  const m = source.match(/title\s*:\s*['"`]([^'"`]+)['"`]/);
  return m ? m[1] : null;
}

/**
 * Extracts all top-level named const exports.
 * Matches: export const Foo: ...
 * Excludes 'default' (re-exports of default are not relevant).
 */
export function extractNamedExports(source) {
  const matches = [...source.matchAll(/^export\s+const\s+(\w+)\s*:/gm)];
  return matches.map(m => m[1]).filter(name => name !== 'default');
}

/**
 * Extracts the keys of the argTypes object block.
 * Only captures the first argTypes block found.
 * Returns keys in source order.
 */
export function extractArgTypesKeys(source) {
  const block = source.match(/argTypes\s*:\s*\{([\s\S]*?)\}/);
  if (!block) return [];
  const keys = [...block[1].matchAll(/(\w+)\s*:/g)];
  return keys.map(m => m[1]);
}

/**
 * Computes a SHA-1 hash for a component entry.
 * Sorts variants and states before hashing so order changes don't alter the hash.
 */
export function computeHash(name, variants, states) {
  const input =
    name +
    JSON.stringify([...variants].sort()) +
    JSON.stringify([...states].sort());
  return createHash('sha1').update(input).digest('hex');
}

// ── File walker ────────────────────────────────────────────────────────────

/**
 * Recursively finds all *.stories.tsx files under a directory.
 */
function findStoryFiles(dir) {
  const results = [];
  let entries;
  try {
    entries = readdirSync(dir, { withFileTypes: true });
  } catch {
    return results; // components/ may not exist yet (plan 1 not run)
  }
  for (const entry of entries) {
    const fullPath = join(dir, entry.name);
    if (entry.isDirectory()) {
      results.push(...findStoryFiles(fullPath));
    } else if (entry.isFile() && entry.name.endsWith('.stories.tsx')) {
      results.push(fullPath);
    }
  }
  return results;
}

// ── Token extraction helpers ───────────────────────────────────────────────

/**
 * Infers which design tokens a component likely uses by scanning for
 * CSS custom property references in the story source.
 * Returns an array of dot-notation token names (e.g. 'color.brand.primary').
 */
function inferTokens(source) {
  const varRefs = [...source.matchAll(/var\(--([a-z0-9-]+)\)/g)];
  return [...new Set(
    varRefs.map(m => m[1].replace(/-/g, '.'))
  )];
}

// ── Main ───────────────────────────────────────────────────────────────────

const storyFiles = findStoryFiles(COMPONENTS_DIR);

if (storyFiles.length === 0) {
  console.warn('[export-manifest] Warning: no *.stories.tsx files found under components/');
  console.warn('  Run plan-1 first to scaffold components, or create at least one story file.');
}

const components = [];

for (const filePath of storyFiles) {
  const source = readFileSync(filePath, 'utf8');

  const fullTitle = extractTitle(source);
  if (!fullTitle) {
    console.warn(`[export-manifest] Skipping ${filePath} — no title found in default export`);
    continue;
  }

  // Component name = last segment of title path (e.g. 'Components/Button' → 'Button')
  const name = fullTitle.split('/').at(-1).trim();

  const namedExports = extractNamedExports(source);
  const argTypesKeys = extractArgTypesKeys(source);

  // Variants = named story exports (these ARE the variants in CSF3)
  const variants = namedExports;

  // States = argTypes keys that represent state dimensions (exclude 'variant' itself)
  // Convention: any argType key that is not 'variant' is treated as a state dimension.
  const states = argTypesKeys.filter(k => k !== 'variant');

  const tokens = inferTokens(source);
  const hash = computeHash(name, variants, states);

  components.push({ name, variants, states, tokens, hash });
}

// Sort components alphabetically for deterministic output
components.sort((a, b) => a.name.localeCompare(b.name));

const manifest = {
  version: '1.0.0',
  generatedAt: new Date().toISOString(),
  components,
};

const json = JSON.stringify(manifest, null, 2);

if (DRY_RUN) {
  console.log(json);
} else {
  writeFileSync(OUTPUT_PATH, json + '\n', 'utf8');
  console.log(`[export-manifest] Written: ${OUTPUT_PATH}`);
  console.log(`[export-manifest] ${components.length} component(s): ${components.map(c => c.name).join(', ')}`);
}
```

### Task 4: Git pre-push hook

**Files:**
- Create: `.githooks/pre-push`

The hook runs `export-manifest.mjs`, diffs the result against the committed version, and auto-commits if changed. This ensures `component-manifest.json` is always up to date on every push without requiring manual steps.

- [ ] **Step 1: Create `.githooks/` directory and the hook file**

Create `.githooks/pre-push` with the following content:

```bash
#!/usr/bin/env bash
# .githooks/pre-push
# Runs export-manifest.mjs before push.
# If component-manifest.json changed, auto-stages and commits it.

set -e

echo "[pre-push] Running export-manifest.mjs..."
node scripts/export-manifest.mjs

# Check if component-manifest.json has uncommitted changes
if git diff --quiet HEAD -- component-manifest.json && git diff --cached --quiet -- component-manifest.json; then
  echo "[pre-push] component-manifest.json unchanged — nothing to commit"
else
  echo "[pre-push] component-manifest.json changed — auto-committing..."
  git add component-manifest.json
  git commit -m "chore(manifest): auto-update component-manifest.json [pre-push]"
  echo "[pre-push] Committed updated manifest"
fi

echo "[pre-push] Done"
exit 0
```

## Gotchas and Notes

**Why regex and not an AST parser?**
The story files follow a consistent, machine-generated pattern from Storybook CSF3. A full AST parser (e.g. TypeScript compiler API or acorn) adds ~5MB of dependencies and complexity with no practical benefit for this structured format. Regex is sufficient, faster, and has zero dependencies.

**Why is `component-manifest.json` committed to the repo?**
n8n cannot execute TypeScript files. The manifest must be pre-built and committed so n8n can fetch it via the GitHub Contents API (same pattern as `tokens/*.json`). The pre-push hook ensures the committed manifest is always fresh.

**Why does the Filter Token Changes node not include `component-manifest.json`?**
The filter exists to prevent spurious n8n executions on unrelated pushes (e.g. README changes). The manifest is regenerated from components, and the Figma plugin fetches it directly from the raw GitHub URL on demand — n8n does not need to be the manifest delivery mechanism. If automatic Figma refresh on component push is desired, that belongs in Plan 3 (Figma plugin sync).

**n8n `Fetch component-manifest.json` input index**
n8n multi-input Code nodes receive inputs by index in the order connections are declared. Input index 3 receives data from `Fetch component-manifest.json`. Inside the jsCode, reference it as `$node['Fetch component-manifest.json'].json.content` (the base64-encoded file content from the GitHub Contents API response).

**`generatedAt` is not part of the hash**
The hash only covers `name + variants + states`. `generatedAt` changes on every run, so it must not affect the hash. The Figma plugin uses the hash — not `generatedAt` — to determine whether to skip or rebuild a component.

**Hook will not fire if `npm run prepare` has not been run**
New clones of this repo must run `npm install` (which triggers `prepare`) before the hook activates. Document this in `FIGMA_SYNC.md` if needed.

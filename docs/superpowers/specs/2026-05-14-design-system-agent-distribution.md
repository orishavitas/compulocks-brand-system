# Design System Agent Distribution
**Date:** 2026-05-14  
**Status:** Approved for implementation

---

## Problem

Every agent (Claude Code, Codex, Claude.ai web, GPT, Gemini) that builds frontend work needs to consume the correct, up-to-date design system — tokens, components, variants, usage rules — without manual copy-paste or guessing. At the same time, the design system must only grow when an **authorized human** explicitly approves it. Agents must never invent tokens or components that haven't been approved.

This system must work as both a **skill** (invocable in any agent session) and an **agent** (autonomous UX/UI prep role), across all platforms the company uses.

---

## Hard Rules

1. **Only named humans on the contributors whitelist can write to the design system.** No agent, ever.
2. **Agents build with stable-only components.** `draft` components do not exist to agents.
3. **Missing component = design request, not improvisation.** Agent logs the gap and parks.
4. **GitHub (`compulocks-brand-system`) is the source of truth.** Everything else is derived.
5. **The living HTML is the source of truth for humans.** Auto-generated, never hand-authored.
6. **All platforms can read. Only the MCP server enforces write authorization.**

## Architecture

### The MCP Server — single access point for all platforms

A local MCP server (`compulocks-design-mcp`) runs on this machine and exposes:

| Tool | Access | Description |
|------|--------|-------------|
| `get_tokens` | Read — all | Returns full tokens.json |
| `get_manifest` | Read — all | Returns manifest (stable components only) |
| `get_spec` | Read — all | Returns SPEC.md content |
| `get_component` | Read — all | Returns single component detail by name |
| `list_components` | Read — all | Lists all stable components with variants |
| `request_component` | Write — agents | Appends to design-requests.md (agents log gaps here) |
| `approve_component` | Write — authorized humans only | Flips status draft→stable, rebuilds, syncs vault |
| `refresh` | Write — authorized humans only | Full rebuild + vault sync |
| `get_requests` | Read — authorized humans only | Lists open design requests |

**Authorization model for write tools:**  
Every write call passes a `contributor_id` (name or email). The server checks against `contributors.json` in the repo root. If not listed → hard reject, logged.

**Platform access:**
- Claude Code / Codex → MCP client (stdio or HTTP, local)
- Claude.ai web → MCP over HTTP (server bound to `localhost`, tunneled or on LAN)
- Other LLMs → same HTTP endpoint, API key gated

### Three artifacts agents read

| File | Contents |
|------|----------|
| `tokens.json` | All token values (colors, type, spacing, radius, shadows) |
| `manifest.json` | Stable components only — name, variants, states, usage notes |
| `SPEC.md` | Human-readable rules — do/don't, dark mode, composition patterns |

All three served via MCP tools AND available at `~/.compulocks/design/` for terminal-native reads.

### The living HTML (human approval surface)

`design-system/index.html` — auto-generated, committed to repo. Contains:
- **Foundations:** color palette, typography scale, spacing, radius, shadows
- **Atoms:** every component with all variants and states, rendered visually
- **Patterns:** common compositions (form, card group, action bar)
- **Icons:** full icon set with names
- **Dark mode:** side-by-side swatches for every token

Generated by `npm run build`. Never hand-edited.

### The contributors whitelist

`contributors.json` in repo root:

```json
{
  "contributors": [
    { "id": "ori@compulocks.com", "name": "Ori", "role": "owner" }
  ]
}
```

Only entries here can call write tools on the MCP server. Adding a contributor = PR to this file.

## The Loops

### Loop 1 — Agent builds a feature (happy path)

```
Agent starts frontend task
  → calls get_manifest() — reads stable components only
  → calls get_tokens() — reads all token values
  → builds using stable components and token CSS variables
  → done
```

### Loop 2 — Agent hits a missing component

```
Agent needs a component not in manifest (or status: draft)
  → calls request_component({ name, reason, usage_context })
      → appended to design-requests.md
  → parks that part of the task
  → completes everything else using stable components
  → flags blocked work clearly in output
```

### Loop 3 — Human-triggered design system refresh

```
Authorized human reviews design-requests.md (or initiates proactively)
  → designs in Figma / Claude Design
  → coder agent implements component + Storybook story (status: draft)
  → human calls: approve_component({ name, contributor_id })
      → server checks contributors.json
      → status flips to "stable" in manifest
      → npm run build triggered
      → index.html regenerated
      → ~/.compulocks/design/ vault updated
      → all agents unblocked instantly
```

## The Skill

A skill (`design-system`) invocable in any Claude Code or Codex session:

```
/design-system          → print current stable components + token summary
/design-system request  → log a design request for a missing component
/design-system status   → show all components with status (human only)
/design-system approve  → approve a draft component (human only, checked)
/design-system refresh  → full rebuild (human only, checked)
```

The skill wraps MCP tool calls — same authorization enforced underneath.

## The Agent Persona (UX/UI Prep Agent)

A named agent role (`ux-prep`) whose job runs before any frontend implementation task:

```
1. Call get_manifest() — load stable component list
2. Call get_tokens() — load token values
3. Call get_spec() — load usage rules
4. Analyze the feature brief
5. Output a "UI prep sheet":
   - Which stable components to use and where
   - Which token variables to use (no hardcoded values)
   - Any gaps → request_component() for each
   - Dark mode considerations
   - Accessibility notes
6. Hand off to coder agent with prep sheet attached
```

This agent runs in all environments via the MCP server — same behavior whether in Claude Code terminal, Claude.ai web, or another LLM.

## Component Status Model

| Status | Meaning | Visible to agents? |
|--------|---------|-------------------|
| `draft` | Implemented, not yet reviewed by human | No |
| `stable` | Human-approved, production-ready | Yes |
| `deprecated` | Flagged for removal | Yes, with warning |

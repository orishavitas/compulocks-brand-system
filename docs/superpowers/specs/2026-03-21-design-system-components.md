# Compulocks Design System — Component Library & Figma Sync

**Date:** 2026-03-21
**Status:** Approved for implementation

---

## Overview

Extend the existing token system into a full design system: React component library published as an npm package, Storybook for documentation and variant definition, automated Figma component library generation, and a structured manifest pipeline connecting git to Figma.

---

## Goals

1. Auto-generate a Figma style guide page and component library page from tokens + Storybook stories
2. Keep Figma in sync with git — skip unchanged components, update changed ones
3. Publish components as `@compulocks/ui` npm package so any product can import brand-correct UI without a designer involved per-product

---

## Non-Goals

- Figma → git direction for components (tokens-only push remains unchanged)
- Fully headless/CI Figma generation (Figma Plugin API requires Desktop — one-click trigger is the ceiling on free tier)
- Linear integration (deferred to future phase)

## Data Flow

### Auto pipeline (git push → manifest)

```
git push
  → GitHub webhook → n8n workflow-a
  → fetch tokens/*.json from GitHub Contents API
  → fetch components/**/*.stories.ts from GitHub Contents API
  → run export-manifest logic (inline Code node)
  → POST manifest JSON to GitHub Gist (public, versioned)
  → existing variable sync continues unchanged
```

### Manual Figma sync (designer-triggered)

```
Figma plugin → "Sync Components" button
  → fetch component-manifest.json from GitHub Gist URL
  → read existing Figma pages
  → ensure "🎨 Style Guide" page exists
  → ensure "🧩 Components" page exists
  → build/update style guide frames (color swatches, type specimens, spacing scale)
  → build/update component frames per manifest
      → skip if component name + variant hash unchanged
      → update if hash changed
      → create if new
```

### Dev consumption (any product)

```
npm install @compulocks/ui
import { Button, Card, Badge } from '@compulocks/ui'
import '@compulocks/ui/styles.css'   # injects CSS custom properties from tokens
```

## Component Manifest Schema

Generated by `scripts/export-manifest.mjs`, committed to `component-manifest.json` at repo root, consumed by n8n and the Figma plugin.

```json
{
  "version": "1.0.0",
  "generatedAt": "2026-03-21T00:00:00Z",
  "components": [
    {
      "name": "Button",
      "variants": ["primary", "secondary", "ghost"],
      "states": ["default", "disabled", "loading"],
      "tokens": ["color.brand.primary", "color.brand.highlight", "font.family.secondary"],
      "hash": "a3f8c2..."
    }
  ]
}
```

`hash` is pre-computed by `export-manifest.mjs` as `SHA-1(name + JSON.stringify(variants.sort()) + JSON.stringify(states.sort()))`. The plugin uses this directly rather than recomputing — it compares `component.hash` from the manifest against the value stored in `node.getPluginData('manifestHash')`.

## Storybook

- Added to this repo via `npx storybook@latest init`
- Stories live alongside components: `components/Button/Button.stories.tsx` (`.tsx` — stories include JSX render functions)
- Each story exports named variants using Storybook CSF3 format with `args` objects
- Storybook serves as both developer documentation and the source of variant truth
- `export-manifest.mjs` does **not** execute or import story files — it statically analyzes them using regex/AST parsing to extract the default export `title`, named exports (variant names), and `argTypes` keys. Output is committed to `component-manifest.json` in the repo root.

## Figma Plugin Changes

### New UI
- Third button: "Sync Components" (alongside existing Pull / Push)
- Status line shows last sync time and component count

### New message handlers in code.ts
- `sync-components` message: receives manifest, builds/updates Figma pages
- Skip logic: for each component in the manifest, compute `SHA-1(name + JSON.stringify(variants.sort()) + JSON.stringify(states.sort()))`. Compare against the value stored via `node.getPluginData('manifestHash')` on the existing `ComponentSetNode`. If equal, skip. If different or absent, rebuild and call `node.setPluginData('manifestHash', newHash)` after. SHA-1 is computed via a small inline implementation (no Node crypto available in plugin sandbox).
- Page management: creates "🎨 Style Guide" and "🧩 Components" pages if absent

### Style Guide page frames
- Color swatches: one frame per token, filled with token value, labelled with name + hex
- Typography specimens: one frame per textStyle, showing sample text with correct font/weight/transform
- Spacing scale: one frame per spacing token, showing a bar of that pixel height with label

### Component Library page frames
- One top-level `ComponentNode` per component (created via `figma.createComponent()` — not a FrameNode, so designers can create instances and publish to a team library)
- Child `ComponentNode` entries for each variant × state combination, grouped into a `ComponentSetNode` via `figma.combineAsVariants()`
- Node names follow Figma variant convention: `variant=primary, state=default`
- Top-level group frame name: component name (e.g. `Button`)

## npm Package

- **Package name:** `@compulocks/ui`
- **Build tool:** tsup (zero-config, ESM + CJS + types)
- **Peer dependencies:** react ≥ 18, react-dom ≥ 18
- **CSS delivery:** single `styles.css` file, injects all token CSS custom properties

- **Publish target:** npm public registry (or scoped private if preferred)

## n8n Workflow Changes (workflow-a)

New step added after existing token fetch:

1. Fetch `component-manifest.json` from GitHub Contents API (raw file, committed to repo root by `export-manifest.mjs` pre-push hook or CI step — n8n does NOT parse TypeScript story files directly)
2. Decode base64 content → parse JSON
3. Merge with token data if needed for enrichment
4. The manifest is now available to the Figma plugin via a stable GitHub raw URL: `https://raw.githubusercontent.com/{owner}/{repo}/master/component-manifest.json`

**No Gist required.** The committed `component-manifest.json` at a stable raw GitHub URL eliminates the bootstrap problem. The plugin UI has a single "Manifest URL" input field (same pattern as the existing webhook URL input) — pre-filled with the known raw GitHub URL. n8n continues to trigger the plugin-side refresh notification via existing webhook mechanism.

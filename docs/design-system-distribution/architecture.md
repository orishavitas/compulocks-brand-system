## Components

### 1. `compulocks-brand-system` repo (source of truth)

The GitHub repo is the authoritative store. Nothing is canonical until it's committed here.

```
compulocks-brand-system/
├── tokens/                    ← DTCG token files (edit these)
├── components/                ← React components + Storybook stories
├── component-manifest.json    ← status: draft | stable | deprecated
├── contributors.json          ← write-access whitelist
├── design-requests.md         ← agent-logged gap requests
├── design-system/
│   └── index.html             ← living HTML reference (auto-generated)
├── mcp-server/
│   ├── index.ts               ← MCP server entry point
│   ├── tools/                 ← one file per tool
│   └── auth.ts                ← contributor check logic
├── scripts/
│   ├── generate-living-html.mjs
│   ├── sync-vault.mjs
│   └── approve.mjs
└── build-tokens.mjs           ← extended to call above scripts
```

### 2. MCP Server (`mcp-server/`)

**Runtime:** Node.js, TypeScript  
**Transport:** stdio (for Claude Code / Codex) + HTTP on port 3333 (for web/other LLMs)  
**Protocol:** Model Context Protocol (MCP) — standard tool definitions

**Auth flow for write tools:**
```
call approve_component({ name: "Modal", contributor_id: "ori@compulocks.com" })
  → server reads contributors.json
  → checks contributor_id is present and role allows writes
  → if not found → { error: "not authorized", logged: true }
  → if found → executes, logs action to design-audit.log
```

**Data served:**
- `tokens.json` — read from `build/json/tokens.json` (generated by Style Dictionary)
- `manifest.json` — read from `component-manifest.json`, filtered to `status: stable`
- `SPEC.md` — read from `token_guide.md` + inline component usage rules

### 3. Living HTML (`design-system/index.html`)

Auto-generated by `scripts/generate-living-html.mjs`. Single self-contained HTML file — no server needed to open it.

**Sections:**
1. **Header** — version, generated date, contributor list
2. **Foundations** — color swatches (all tokens), typography scale specimens, spacing bars, radius demos, shadow demos
3. **Atoms** — each component rendered in all variants × states. Draft components shown with a warning banner — not available to agents.
4. **Patterns** — common compositions using stable atoms (form layout, card group, action bar, data table row)
5. **Icons** — full icon set grid with token names
6. **Dark mode** — every foundation token shown in light/dark side-by-side

**Human approval interaction:** a small sticky toolbar at the top shows pending drafts. Click component name → jumps to its section. "Approve" button calls `npm run approve <name>` instruction (copy-paste ready command shown in UI).

### 4. Vault (`~/.compulocks/design/`)

Local directory. Populated by `scripts/sync-vault.mjs` on every successful build. Agents that can't reach the MCP server (offline, no port access) fall back to reading these files directly.

```
~/.compulocks/design/
├── tokens.json          ← full token values
├── manifest.json        ← stable components only
├── SPEC.md              ← usage rules
└── .last-updated        ← ISO timestamp of last sync
```

### 5. Skill (`/design-system`)

A Claude Code skill at `.claude/skills/design-system.md`. Wraps MCP tool calls with a clean interface:

```
/design-system              → list stable components + token summary
/design-system request      → log a design request (agent-safe)
/design-system status       → all components + status (humans only)
/design-system approve      → approve draft component (humans only)
/design-system refresh      → full rebuild + vault sync (humans only)
```

### 6. UX-Prep Agent Persona (`agents/ux-prep.md`)

An agent role invokable in any session. Runs as a mandatory pre-step before any frontend implementation task.

**Inputs:** feature brief or task description  
**Process:** calls `list_components`, `get_tokens`, `get_spec` → analyzes brief → maps UI elements to design system  
**Output:** a UI prep sheet attached to the task:
- Component map (which stable component → which UI element)
- Token variables to use (no hardcoded values permitted)
- Design requests filed for any gaps
- Dark mode notes
- Accessibility flags

## Data Flow — New Component Lifecycle

```
1. DESIGN     Figma / Claude Design → visual spec exists
2. IMPLEMENT  Coder agent → component.tsx + component.stories.tsx (status: draft)
3. BUILD      npm run build → index.html regenerated, draft visible with warning
4. REVIEW     Human opens design-system/index.html → inspects visually
5. APPROVE    Human: npm run approve ComponentName
                → contributors.json checked
                → manifest status: draft → stable
                → npm run build (auto-triggered)
                → vault synced
                → MCP server hot-reloads manifest
6. AVAILABLE  All agents — all platforms — can now use the component
```

## Authorization Matrix

| Action | Agent (any) | Unauthorized human | Authorized human |
|--------|-------------|-------------------|-----------------|
| Read tokens | ✅ | ✅ | ✅ |
| Read stable manifest | ✅ | ✅ | ✅ |
| Read spec | ✅ | ✅ | ✅ |
| File design request | ✅ | ✅ | ✅ |
| Read all requests | ❌ | ❌ | ✅ |
| Approve component | ❌ | ❌ | ✅ |
| Trigger refresh | ❌ | ❌ | ✅ |
| Modify contributors.json | ❌ | ❌ | ✅ (via PR) |

## Failure Modes

| Failure | Behavior |
|---------|----------|
| MCP server not running | Agent falls back to reading vault files directly |
| Vault not populated | Agent surfaces clear error: "Design system vault not found. Run npm run build in compulocks-brand-system." |
| Unauthorized write attempt | Hard reject + entry in design-audit.log |
| Component requested but not yet approved | Stays draft, design-requests.md updated, agent parks |
| Living HTML out of date | index.html has a "last generated" timestamp — human sees if it's stale |
